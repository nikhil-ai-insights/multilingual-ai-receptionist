## 🚀 Getting Started

### Step 1 — Fork & Clone

```bash
# Fork the repo via GitHub UI, then clone your fork
git clone https://github.com/your-username/multilingual-ai-receptionist.git
cd multilingual-ai-receptionist
```

### Step 2 — Install Dependencies

```bash
npm install
```

### Step 3 — Create a Branch

> ⚠️ Never commit directly to `main`. Always work on a dedicated branch.

```bash
git checkout -b feature/your-feature-name
```

**Branch naming conventions:**

| Type | Example |
|---|---|
| New feature | `feature/calendar-sync` |
| Language support | `feature/hindi-support` |
| Bug fix | `fix/login-bug` |
| UI change | `ui/new-dashboard` |
| Docs update | `docs/api-reference` |

## 📌 Development Guidelines

### Code Quality Standards

| Principle | What it means |
|---|---|
| ✅ **Clean** | Easy to read and follow |
| ✅ **Modular** | Small, reusable components/functions |
| ✅ **Scalable** | Designed to grow without rewriting |
| ✅ **Consistent** | Follows existing project structure |
| ✅ **Documented** | Key logic has clear comments |

### Frontend
- Fully responsive across screen sizes
- Consistent spacing, typography & colors
- Accessible UI (WCAG-friendly where possible)
- Reusable components

### Backend
- RESTful API design
- Input validation on all endpoints
- Proper error handling & status codes

- Optimized database queries

## 🧪 Pre-Submission Checklist

Before opening a Pull Request, confirm the following:

```
[ ] Code runs without errors
[ ] No build or lint issues
[ ] UI renders correctly (if frontend change)
[ ] APIs tested manually or with tests
[ ] Existing features are not broken
[ ] Commit messages follow the format below
```

---

## 📝 Commit Message Format

Use clear, structured commit messages:

```
<type>: <short description>
```

| Type | When to use |
|---|---|
| `feat` | Adding a new feature |
| `fix` | Fixing a bug |
| `ui` | Visual / design changes |
| `docs` | Documentation updates |
| `refactor` | Code restructure (no behavior change) |
| `test` | Adding or updating tests |
| `chore` | Config, build, or maintenance tasks |

**Examples:**
```bash
feat: add Google Calendar integration
fix: resolve login authentication issue
ui: redesign dashboard booking cards
docs: update installation guide with env setup
refactor: improve appointment booking service logic
```

## 🔄 Pull Request Process

### 1. Push Your Branch
```bash
git push origin feature/your-feature-name
```

### 2. Open a Pull Request on GitHub

Include the following in your PR description:

```markdown
## What does this PR do?
Brief summary of changes.

## Type of change
- [ ] Bug fix
- [ ] New feature
- [ ] UI/Design update
- [ ] Documentation
- [ ] Refactor

## Screenshots (if UI change)
Add before/after screenshots here.

## Testing done
Describe how you tested your changes.
```

### 3. Review & Merge

- A maintainer will review your PR
- Feedback or revision requests may be made
- Once approved, your PR will be merged 🎉

## 🐛 Reporting Bugs

Found something broken? [Open an Issue](https://github.com/nikhil-ai-insights/multilingual-ai-receptionist/issues/new) with the following:

```

**Expected behavior:** What should happen?

**Actual behavior:** What actually happens?

2. Click '...'
3. See error

**Environment:**
- OS: [e.g. macOS 14, Windows 11]
- Browser: [e.g. Chrome 124]
- Node version: [e.g. v18.17]

**Screenshots:** (attach if helpful)
```
